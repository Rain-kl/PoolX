

需求文档
Clash / Mihomo 代理池控制端（兼容多内核扩展）
项目主题：Clash / Mihomo 代理池控制端，兼容多内核扩展
版本：v1.0（整合稿）
定位：Proxy Kernel Control Plane

说明：本稿已将“Clash 控制端”与“多内核统一控制平台”两份内容合并。当前版本以 Mihomo 为首个可落地内核，同时保留 sing-box / Xray 的扩展抽象。


1. 文档目的
   本文档定义项目背景、目标、范围、角色、流程、功能与非功能需求，以及验收标准。
2. 项目定位
   • 产品正式定位为 Proxy Kernel Control Plane，而非单一 Clash 控制端。
   • MVP 首期仅落地 Mihomo 适配器，前后端模型不绑定 Mihomo 专有字段。
   • 平台负责导入节点、管理节点池、创建监听入口、配置策略、渲染配置、控制内核进程与展示状态日志。
   • 平台不自行实现代理协议和流量转发，也不让前端直接访问内核私有 API。
3. 目标与成功标准
   目标	说明	成功标准
   配置导入	上传并解析 YAML/后续可扩 JSON	成功解析有效节点，错误可定位
   节点可用性验证	支持 TCP / HTTP / URL Test	批量测试可并发、可追踪进度、结果可缓存
   节点池管理	统一检索、筛选、启停与去重	用户可快速找到可用节点并形成池
   工作台配置	按端口配置监听与策略	支持保存、预览、版本化生成运行配置
   核心控制	启动 / 停止 / 热重载 Mihomo	状态准确、日志可见、异常可诊断
   未来扩展	支持 sing-box / Xray 适配	不改业务主链路，仅新增适配器


4. 业务流程
   • 流程 1：导入配置文件 → 解析节点 → 去重与标准化 → 预览结果 → 执行节点测试 → 用户确认导入节点池。
   • 流程 2：从节点池选择节点 → 创建端口配置 → 选择监听协议、策略、测试 URL 与间隔 → 保存工作台配置。
   • 流程 3：系统基于数据库结构化数据生成运行配置 → 生成快照 → 启动或热重载 Mihomo → 展示状态与日志。
   • 流程 4：未来切换目标内核时，仅切换适配器与渲染器，业务流程保持不变。
6. 功能范围
   模块	MVP 必须	后续增强
   配置导入	上传 YAML、解析、预览、确认	支持订阅源、JSON 格式
   节点测试	TCP / HTTP / URL Test	首包耗时、自动刷新
   节点池	搜索、筛选、批量启停、测试结果	标签体系、智能分组
   工作台	端口 CRUD、节点选择、策略选择	模板保存、工作区复制
   运行控制	启动 / 停止 / 热重载 / 状态 / 日志	多实例、容器化部署
   内核扩展	能力协商模型预埋	sing-box / Xray 适配器

7. 功能需求
   7.1 配置导入
   • 支持 .yaml / .yml 上传并做大小限制10M。
   • 解析 proxies 字段并容忍非关键字段缺失。
   • 按 name + server + port + type 与配置签名双层去重。
   • 保留结构化字段与原始配置，便于检索与回生成配置。
   7.2 节点测试
   • 支持批量并发测试，单批建议 50~100 并发。
   • 单节点超时 3~8 秒，失败最多重试 1 次。
   • 提供基于内核实际代理链路的 URL Test，避免仅 TCP 假阳性。
   • 通过 WebSocket/SSE 返回任务进度与结果。
   7.3 节点池管理
   • 支持分页、搜索、筛选、启用/禁用、最近测试结果展示。
   • 支持按协议、地区、延迟区间、来源配置等维度筛选。
   • 支持去重合并与后续标签扩展。
   7.4 工作台配置
   • 一个端口配置代表一个独立入口。
   • 可配置监听地址、端口、入站类型、节点集合、策略、测试 URL 与间隔。
   • 策略对用户展示统一概念：随机、负载均衡、故障转移、延迟优先、手动。
   • 保存后必须可预览生成配置。
   7.5 运行控制
   • 支持生成运行配置、启动、停止、热重载、状态查询、日志查看。
   • 前端不得直接调用内核 API，统一由后端代理。
   • 配置变更时需要判断哈希，未变化则不触发重载。
8. 非功能需求
   类别	要求
   安全	Mihomo API 默认仅监听 127.0.0.1；使用 secret 鉴权；上传文件大小、层级与字段白名单限制；不得拼接 shell。
   可靠性	配置全部由数据库生成；保存快照与版本号；启动与重载前做端口占用和健康检查。
   性能	节点测试结果缓存 1~5 分钟；列表查询分页；前端支持虚拟滚动。
   可观测性	记录解析、导入、测试、渲染、启停、健康检查日志；提供最近错误与 stderr 查看入口。
   可扩展性	引入内核能力协商与适配器模式，确保未来扩展 sing-box / Xray 时无需重写主流程。

9. 约束与关键取舍
   • MVP 阶段采用单用户单运行实例单配置，降低实现复杂度。
   • 随机策略只作为产品语义存在，在核心层映射为 load-balance/round-robin 等等价实现。
   • 不开放复杂规则编辑器，MVP 仅做 global 模式与最小化规则。
   • 不允许用户直接编辑运行中的唯一配置文件，必须由数据库重新生成。
   • 对不同内核能力差异采用能力降级提示，而非前端写死功能。
10. 验收标准
	• 上传一个合法 Clash/Mihomo YAML 后，可返回解析摘要、有效节点列表和错误列表。
	• 用户确认导入后，节点可在节点池中查询、筛选并查看最近测试结果。
	• 用户至少可创建一个端口配置，选择多个节点与策略，保存后能预览生成配置。
	• 可成功启动 Mihomo，状态页能显示进程状态、监听端口、API 健康状态与日志流。
	• 配置更新后若哈希变化，可执行热重载并保留运行配置快照。
	• 前端可根据能力模型渲染当前内核支持的策略和操作按钮。
11. 里程碑
	阶段	交付内容
	Phase 1	上传 YAML、解析节点、节点列表、节点测试、导入节点池
	Phase 2	端口配置 CRUD、节点选择器、策略选择器、配置生成与预览
	Phase 3	启停 Mihomo、热重载、运行状态与日志展示
	Phase 4	测速优化、节点标签、模板保存、自动刷新、能力协商完善

结论：本项目需求以“统一控制平面、MVP 先落地 Mihomo”为主线，后续通过内核适配器扩展到更多代理核心。
