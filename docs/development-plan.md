# Proxy Kernel Control Plane 开发计划

本文档定义当前业务主线下的默认迭代顺序、阶段目标和准入规则。

当前规划基于以下前提：

* 以 `PoolX` 现有基础设施为底座
* MVP 首期只交付单用户、单实例、单运行配置的 Mihomo 闭环
* 前后端模型采用统一产品概念，不直接绑定 Mihomo 私有结构
* 工程继续沿用当前 `server/internal/{handler,service,model,...}` 分层

## 1. 总体策略

开发顺序遵循：

1. 先打通数据模型与服务骨架，再交付页面
2. 先完成“可保存、可预览、可验证”，再做“可启动、可重载”
3. 先完成 Mihomo 单内核闭环，再补能力协商与多内核扩展点
4. 每一阶段都要求前后端、接口、导航、Swagger、测试与文档同步落地

## 2. 阶段计划

### Phase 0 文档与基线对齐

目标：

* 把项目主线从“纯模板工程”切换为“基于模板承载代理池控制端业务”
* 明确当前采用的分层方式、MVP 边界和实施顺序

交付物：

* `AGENTS.md`
* `docs/design.md`
* `docs/development-guidelines.md`
* `docs/frontend-development-guidelines.md`
* `docs/development-plan.md`
* `README.md`

准入结果：

* 后续开发不再与模板边界文档冲突

### Phase 1 基础模型与导入链路

目标：

* 建立配置导入、节点池和节点测试的最小可用后端闭环
* 搭好后续工作台与运行控制要复用的数据结构

服务端范围：

* 设计并迁移 `source_configs`、`proxy_nodes`
* 建立 YAML 上传、解析、标准化、双层去重、确认导入服务
* 建立节点分页查询、搜索筛选、启用禁用接口
* 建立节点测试任务模型、结果存储与查询接口
* 为测试进度预留 SSE / WebSocket 输出能力

前端范围：

* 新增配置导入页
* 新增节点池页
* 新增基础导航与 API client
* 完成解析摘要、节点表格、测试结果反馈

完成标准：

* 上传合法 YAML 后，可返回解析摘要、有效节点与错误信息
* 用户确认导入后，节点可在节点池中分页查询和筛选
* 节点测试结果可回写并展示最近状态

### Phase 2 工作台与配置预览

目标：

* 建立从节点池到端口配置、再到运行配置预览的主链路

服务端范围：

* 设计并迁移 `port_profiles`、`port_profile_nodes`、`runtime_configs`
* 建立端口配置 CRUD、节点绑定、策略配置接口
* 建立 Runtime IR 构建器
* 建立 Mihomo 渲染器 MVP
* 建立配置预览与校验和生成逻辑

前端范围：

* 新增工作台页
* 实现端口配置表单、节点选择器、策略选择器、预览抽屉
* 建立能力协商结果对应的策略选项降级提示

完成标准：

* 用户至少可创建一个端口配置并绑定多个节点
* 保存后可查看生成配置预览
* 配置快照具备版本与校验和

### Phase 3 运行控制闭环

目标：

* 完成 Mihomo 单实例启动、停止、重载、状态查询与日志查看

服务端范围：

* 设计并迁移 `kernel_instances`
* 建立 Mihomo 启动器、停止器、重载器、状态读取器、健康检查器
* 落地“校验 -> 构建 IR -> 渲染 -> 写文件 -> 计算校验和 -> 启动/重载”的统一流程
* 建立运行状态、日志流、最近错误接口
* 建立配置未变化时跳过重载的机制

前端范围：

* 新增运行状态页
* 展示进程状态、监听端口、API 健康、日志流与最近动作结果
* 补齐 Dashboard 基础指标卡

完成标准：

* 可成功启动 Mihomo
* 状态页可看到运行状态、日志与健康信息
* 配置变化后可执行热重载，未变化时不会误触发

### Phase 4 稳定性与能力协商增强

目标：

* 完善可用性、反馈质量和未来扩展点

服务端范围：

* 完善能力协商模型与接口
* 完善测试缓存、错误分类、重试策略与健康检查
* 为 sing-box / Xray 预留适配器注册点
* 增加关键链路单元测试与回归测试

前端范围：

* 完善 Dashboard 指标、导入与测试反馈、运行状态过滤与空态
* 完善基于能力协商的按钮启用、禁用与提示

完成标准：

* Mihomo MVP 主链路具备稳定回归保障
* 前端不再写死 Mihomo 能力表达

## 3. 当前建议实施顺序

默认按以下顺序推进：

1. 先落 Phase 1 的数据模型、服务、接口和最小页面
2. 再落 Phase 2 的工作台、IR 与渲染预览
3. 最后落 Phase 3 的运行控制与日志流
4. Phase 4 在主链路跑通后插入做稳定性补强

如果中途要并行开发，也必须保证依赖顺序不被破坏：`导入/节点 -> 工作台/IR -> 运行控制`。

## 4. 准入规则

每个阶段合入前至少满足：

* 后端接口、前端页面、导航、类型、Swagger 同步完成
* 迁移脚本、空库初始化与旧库升级路径可执行
* 服务端通过 `go test ./...` 或等效验证
* 前端通过 `pnpm build`，必要时补充组件测试或端到端测试
* 文档与当前实现一致，不保留明显失效入口

## 5. 当前风险与决策

当前已知风险：

* `docs/dev/Tech.md` 提出的推荐目录结构与仓库现状不一致，若直接照搬会放大重构成本
* 运行控制、日志流和节点测试都属于高副作用能力，必须先建立清晰边界再编码
* 多内核扩展是中长期目标，MVP 阶段不能为了“未来通用”牺牲当前交付节奏

当前决策：

* 保留现有 MVC 分层，只吸收统一模型、IR、Adapter 思路
* 先实现 Mihomo 单实例闭环，再补多内核
* 先把数据库作为唯一真实来源，再生成运行配置，不开放手工直接改运行文件
